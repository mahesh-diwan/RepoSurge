## 3. Cognitive Load

- [x] **Chunking**: Repos listed as rows, information grouped naturally (rank, name, sparkline, gained)
- [x] **Grouping**: Header row groups sort controls. Period selector is visually separated. Stats in hero are grouped

- [x] **One thing at a time**: Users can focus on the list without competing tasks
- [x] **Minimal choices**: 3 sort options, 4 period views, search — all ≤4
- [x] **Working memory**: List data is self-contained. No need to remember across pages

## 4. Emotional Journey

The experience is flat-positive. Landing → seeing the ranked list → live delta updates provide a mild pleasure sequence. No emotional valleys — there are no destructive actions, no error-prone forms.

**Peak-end:** The peak is seeing live star counts increment in real-time (the green `+N` deltas + "live" badge are the most emotionally engaging element). The end (footer, back-to-list) is neutral. No emotional letdown.

**High-stakes moments:** None. Read-only data browsing. Reassurance is not needed.

**Risk:** The flat emotional arc means nothing delights. The app works, but there's no "wow" moment beyond the initial green-on-black aesthetic.

## 5. Strengths

1. **Cohesive identity.** The terminal-green-on-midnight palette is committed, not decorative. JetBrains Mono, flat rows, `>` prompt in the search bar — every detail reinforces the terminal metaphor without being kitschy. This is the hardest thing to pull off in design and it's done well.

2. **Live polling UX.** The green dot with glow + "live" label + per-repo deltas (+1593, +1058) is instantly legible. Users see motion and know the data is current. The delta timing is unobtrusive (superscript text, not a modal or toast).

3. **Restraint in layout.** No cards, no shadows, no gradients. For a data-heavy tool, the flat-row approach maximizes scanability. The mobile experience preserves the same visual grammar, which is rare.

## 6. Priority Issues

### P1: Low contrast on ALL secondary text (#666 on #0A0A0A = 3.4:1)

- **What:** `text-dim` is `#666` in tailwind.config.ts. On `#0A0A0A` background, this gives a 3.4:1 contrast ratio — below the 4.5:1 WCAG AA minimum. The detector found ≈15 instances across the app.
- **Why it matters:** Secondary text ("gained", "period:", "daily", "live", footer text, rank numbers, about page body) is hard to read, especially on lower-quality monitors or in bright ambient light. Fails WCAG AA, a legal/accessibility risk.
- **Fix:** Bump `dim` to `#888` or `#999` (≈5:1 on `#0A0A0A`). Best: `#999` (≈5.3:1) preserves the subdued feel without failing contrast.
- **Suggested command:** `/impeccable colorize` — adjust the `dim` token in tailwind.config.ts

### P1: Missing heading hierarchy — no semantic headings anywhere

- **What:** The about page has zero `<h1>`–`<h3>` tags. Content is marked up as `<p>` only. The homepage has an `<h1>` (REPØSURGE) but summary stats and section labels use `<p>` tags. Detail pages have the repo name as `<h1>` but period tabs and stat labels lack heading structure.
- **Why it matters:** Screen readers (Sam's persona) cannot navigate the page by headings. Every page is a flat text wall. SEO is also degraded — no heading structure for content discovery.
- **Fix:** About page: wrap "what is this", "how it works", "stack" in `<h2>` tags. Homepage: the hero stats section needs an `<h2>`. Period picker should be `<nav>` with proper labeling.
- **Suggested command:** `/impeccable audit` — accessibility pass, or `/impeccable harden`

### P2: Repo name text-overflow on mobile (detected: 17–101px overflow)

- **What:** The `truncate` class on repo name links doesn't prevent overflow when the sparkline column is hidden (mobile) and the name is long. Detected overflow spans 17–101px.
- **Why it matters:** Long repo names break the layout on mobile, pushing `gained` values off-screen or overlapping. The `truncate` utility has no explicit `max-width` or `overflow: hidden` context to bind to.
- **Fix:** Add a `max-width` (e.g. `max-w-[160px]` or `max-w-[50vw]`) to the repo name link, or ensure the parent flex container distributes space correctly with flex-basis utilities.
- **Suggested command:** `/impeccable adapt` — mobile layout pass

### P2: No active nav state on desktop

- **What:** Desktop nav items (`HOME`, `DAILY`, `WEEKLY`, etc.) have no visual indicator for the current page. All links display as `text-dim` regardless of which page you're on.
- **Why it matters:** Users lose spatial orientation. "Where am I?" requires reading URL or page content. Jordan (First-Timer) gets disoriented. Alex (Power User) finds it inefficient.
- **Fix:** Add an `aria-current="page"` or active class. Highlight the matching link with `text-terminal` instead of `text-dim`.
- **Suggested command:** `/impeccable harden`

### P3: Empty search state lacks guidance

- **What:** Searching "vercel" shows "no repos match vercel" — a functional message but no suggestions, no similar-repo hints, no "try a different name" guidance.
- **Why it matters:** Riley (Stress Tester) notices the dead-end. Jordan doesn't know what to do next. A search box that says "try typing something else" adds friction.
- **Fix:** Add suggestions below the empty message: popular repo names, or a subtle "search by owner name" hint.
- **Suggested command:** `/impeccable clarify`

## 7. Persona Red Flags

### Alex (Power User) — Impatient, efficiency-driven

- **No keyboard shortcuts.** Can't press `s` to focus search, `1-4` for periods, or `/` to start filtering. Every action is a click or tap.
- **Sort toggling is subtle.** Triple-click to clear sort (asc → desc → null) is discoverable only by experimentation. No "reset sort" button.
- **No URL-based state.** Search query and sort are reset on page navigation. Alex can't bookmark a sorted view or share a filtered list.
- **No bulk actions.** No way to compare repos, no way to export the list. Info is consumed or lost.

**Risk:** Moderate. Git star trackers attract technical users who expect keyboard shortcuts and fast filtering. If the data is useful enough, Alex stays; otherwise bounces.

### Casey (Distracted Mobile User) — One-handed, interrupted

- **Search + period picker at top.** On 375px viewport, both controls are in the top third of the screen — hard to reach with one hand.
- **Sparklines hidden on mobile** (`hidden sm:block`) — correct decision. But the text-overflow issue means long names break the layout.
- **Hamburger works well** — backdrop-blur overlay, smooth animation, items centered. The `✕` button is in the same position as `☰`, which is good.
- **No state persistence** — if interrupted while scrolling, returning from another app (or tab switch) the scroll position is lost. Minor for a list view.

**Risk:** Low. The mobile experience is functional and clean. The main pain point is the text-overflow breakage.

### Sam (Accessibility-Dependent) — Screen reader, keyboard-only

- **FAIL: Low contrast** on ALL dim text (3.4:1 vs required 4.5:1). P1 blocker.
- **FAIL: No heading hierarchy.** About page is entirely `<p>` tags. Screen readers have no navigation landmarks.
- **PASS: Focus indicators.** `:focus-visible` is styled with `1px solid #00FF41` and 2px offset — clear and terminal-theme appropriate.
- **PASS: Keyboard nav.** All nav links, sort buttons, and repo links are `<a>`/`<button>` elements — keyboard accessible.
- **PASS: Selection colors.** `::selection` blocks are styled — green bg on black text, visible.
- **PASS: ARIA labels.** Search has `placeholder`, mobile menu has `aria-label="Toggle menu"`, sparkline has `aria-label="star history"` and `role="img"`. Good.

**Risk:** High due to contrast and heading issues. These are WCAG AA failures, not nice-to-haves.

## 8. Minor Observations

1. **Hero stats feel disconnected.** "11 repos · 1,272,000 stars · 4 languages" above the h1 is a strange information flow. The tagline "repos rising. fast." sits above the title. Reversing the h1/tagline or grouping stats below would feel more natural.

2. **Period selector shows "weekly" as unlinked text** on the homepage (current period). This is fine functionally but visually blends into the nav. The `text-terminal font-bold` style for the active period is good — apply the same pattern to desktop nav links.

3. **Footer "updated 5h ago" precision is a nice touch.** The `LastUpdated` component adds a human-readable recency signal. The tooltip-style format works well.

4. **About page body text is `text-dim` (#666)** — same contrast issue. The entire about page content is hard to read.

5. **"repo not found" page** for `/repo/vercel-next-js` (slugs use `-` not `/` in URL) is correct behavior but the error message is terse. A list of available repos would help.

6. **No favicon visible in tab.** The snapshot shows `REPOSURGE - repos rising. fast.` in the title, but no custom favicon rendered (the layout references `/favicon.svg`).

## 9. Provocative Questions

1. **What if the repo rows were clickable anywhere** instead of only the name link? The full row has `group-hover` style on the link, but clicking the rank number or gained value does nothing. A trivial ergonomic gain.

2. **What if the search box accepted GitHub-specific syntax?** `owner:vercel`, `lang:rust`, `stars:>10000` — treating the terminal aesthetic as a genuine query-language affordance rather than decoration.

3. **What if period navigation used swipe gestures on mobile?** Horizontal swipe between daily/weekly/monthly would feel native and reduce the top-of-screen reach zone for Casey.
